# ETL Bidirectional — Feasibility Analysis

*Task 44 output. Last updated: 2026-05-01.*

When a tool table is generated by an ETL SQL query, user modifications to that table can
sometimes be reflected back into the saved SQL. This document maps each table operation
to its SQL correspondence and implementation status.

---

## Operation → SQL mapping

| Operation | SQL analog | Bidirectional feasible? | Auto or confirm? | Status |
|-----------|-----------|------------------------|-----------------|--------|
| Delete ETL column | Remove SELECT expression | Yes | Confirm dialog | **Task 45 — done** |
| Rename ETL column | Rename `AS` alias | Yes | Confirm dialog | **Task 45b — done** |
| Apply visual transform (prefix/suffix/replace) | Wrap expression: `'prefix' \|\| col` | Partial (limited SQL patterns) | Confirm dialog | Task 46 (planned) |
| Add user column | `SELECT *, null AS new_col` | Partial (adds a NULL expression) | Auto | Future |
| Reorder columns | Reorder SELECT list | Yes | Auto | Future |
| Edit cell in ETL column | Handled by `_overrides` (per-cell override) | N/A — already done | — | Done |
| Add / delete row | No SQL analog — ETL owns row lifecycle | No | — | Out of scope |
| Filter rows | WHERE clause — requires row identity mapping | No (row identity problem) | — | Out of scope |

---

## Detection mechanism

A column is ETL-generated if `lineage_info IS NOT NULL` in `_columns`. The field stores
`{"source_expr": "il.service", "from_tool": "instrument_list"}` and is written by
`etl_apply` via `sql_parser.extract_col_lineage` + `lineage_to_source`.

There is no explicit `is_etl_generated` flag — `lineage_info` nullability is the signal.

---

## Auto-update vs confirm

Operations that permanently alter the saved ETL SQL must be confirmed by the user (confirm
dialog). The confirmation message surfaces the ETL source so the impact is clear.

Silent auto-update is acceptable only for additive, easily reversible operations (e.g.,
reordering SELECT columns) where no data is lost.

---

## SQL manipulation strategy

All SQL mutations use regex + depth-0 splitting (see `engine/sql_parser.py`). No
third-party SQL AST library is introduced — the existing `clean_sql` + `_depth0_split`
infrastructure is sufficient for the narrow operations in scope (add/remove/rename a
SELECT item).

Limitation: expressions containing sub-SELECTs with their own FROM clause may confuse
the `SELECT … FROM` boundary regex. These are not common in this codebase's ETL patterns.

---

## ETL save path for auto-updates

Column deletions update `query_config.etl_sql` and `query_config.etl_deps` **without**
adding a history entry (silent draft write, same semantics as `etl_save_draft`). This
avoids polluting the user's version history with system-generated entries.

Both the `UPDATE _tools` (config) and `DELETE _columns` writes are committed in a single
`conn.commit()` in `service.delete_column` — the update is atomic.
